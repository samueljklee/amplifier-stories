Discovery #2

The "Always Finds Something" Problem

Our validator would find issues in every config‚Äîeven exemplary ones. It created distrust and "validator fatigue."

The Symptom
Run validator on our best, most carefully crafted configs.

Result: "3 warnings, 2 suggestions"

Users learned to ignore the validator entirely.
The Root Cause
We asked AI to "find issues and suggest improvements."

AI always finds something to improve.

No concept of "good enough" ‚Üí endless nitpicking.
"A validator that always finds issues is worse than no validator at all."
The Solution

Explicit PASS Thresholds

Define what "good enough" means before AI analysis. Let code decide quality, not AI interpretation.

Quality Tier Criteria (Deterministic) AI Analysis?
‚úì Good Has required fields + meets length thresholds + has examples Skip entirely
‚óê Polish Missing one optional element Light review
‚úó Needs Work Missing multiple elements or too short Full analysis
‚ö† Critical Structural errors (invalid syntax, missing required fields) Error report
The Pattern: Classify quality with deterministic code FIRST. Only invoke expensive AI analysis when it's actually needed.
The Secret Ingredient

Tell AI what NOT to flag

Explicit guidance about what's acceptable prevents nitpicking.

‚ùå Before
"Analyze this configuration and identify any issues, improvements, or suggestions."
Result: 47 suggestions for a perfectly good config
‚úÖ After
"This config passed quality thresholds.

Do NOT flag: style preferences, minor wording, optional fields that are legitimately optional.

Only flag: actual errors or significant gaps."

The Architecture

Deterministic First, AI Second

1. Parse config # Structural validation (code)
2. Check required fields # ERROR if missing (code)
3. Measure quality metrics # Length, completeness (code)
4. Classify into tier # Good/Polish/NeedsWork (code)
5. If tier == "Good": PASS # Skip AI entirely
6. Else: AI analysis # With "what NOT to flag" guidance
Why this works: Code-based classification is consistent and fast. AI is reserved for nuanced judgment where it adds value‚Äînot for gatekeeping good work.

The Takeaways

The 50% Rule (and friends)

üîç The 50% Rule
Assume half of AI-generated rules need verification against actual code. Build verification into your process, not as an afterthought.
‚úì The PASS Path Rule
Every validator needs an explicit "good enough" threshold. If it always finds issues, users will ignore it entirely.
üìã The Deterministic First Rule
Use code for classification and thresholds. Reserve AI for nuanced judgment where it actually adds value.
üìù The Source Fix Rule
Wrong documentation creates wrong AI. When you find errors, fix both the validator AND the docs it learned from.
The Bottom Line

Build fast. Verify ruthlessly.
Define "good enough."

AI accelerates development, but human judgment catches the errors that matter.
