Recipe Engineering at Scale: From Crashes to 354 Sessions

THE MISSION

Bulk Session Analysis

Goal: Process hundreds of Amplifier sessions to understand team workflows, capabilities, and project patterns.

The Challenge: Build a recipe system that could reliably analyze sessions at scale without crashing, hanging, or taking forever.

PROBLEM #1

Death by API Calls

Original Approach

100+ GitHub API calls per session

  • Fetch session metadata
  • Download events.jsonl
  • Download transcript
  • Parse, analyze, repeat

Result: Painfully slow. Hours for large batches.

Optimized Solution

Direct filesystem reads

  • Read local session files
  • Zero API calls
  • Parse directly from disk
  • Orders of magnitude faster

Result: Fast enough to process 354 sessions.

THE BREAKTHROUGH

bulk-analyze-sessions-local.yaml

New recipe that optimizes for local filesystem access:

# Read directly from ~/.amplifier/sessions/ for session_dir in sessions/* read transcript.md read events.jsonl extract capabilities, projects, tools build indexes end # No GitHub API calls required

Performance transformation: From hours to minutes.

PROBLEM #2

Recipe Crashes on Missing Files

The sync-session.yaml recipe would crash when sessions lacked events.jsonl files:

# Original approach - crashes on missing files EVENTS=$(cat ${SESSION_DIR}/events.jsonl) # ❌ Template evaluates BEFORE conditional check # ❌ $EVENTS is empty → recipe crashes

The Investigation: Why do conditionals fail to protect against missing data?

THE INVESTIGATION

Template Variable Evaluation Timing

Critical discovery: Recipe template variables evaluate before conditional checks.

❌ What We Thought
Conditionals protect against missing data. If a file doesn't exist, the conditional skips that step.
✓ What Actually Happens
Template engine evaluates ALL variables first, then checks conditionals. Missing files cause crashes before conditionals run.
THE SOLUTION

The "Option A" Pattern

Key insight: Move conditionals inside bash commands, not in recipe logic.

# ✓ Option A: Conditional inside bash if [ -f "${SESSION_DIR}/events.jsonl" ]; then cat "${SESSION_DIR}/events.jsonl" else echo "" # Always return output fi # Result: No crashes, reliable evaluation # Every step returns output, even if empty

Golden Rule: All recipe steps must always produce output, even if it's an empty string.

BONUS FIX

ARG_MAX and Large Files

Discovered another crash: Large transcripts (580KB+) exceeded shell argument limits.

Before

Passing data via arguments

curl -X POST --data "$TRANSCRIPT"

❌ Fails on large files (ARG_MAX limit)

After

Piping via stdin

echo "$TRANSCRIPT" | curl --data-binary @-

✓ Handles 580KB+ files reliably

Testing verified: Small files (151 bytes) ✅ • Large files (580KB → 218KB base64) ✅

KEY TAKEAWAYS

Recipe Engineering Best Practices

1. Template Variables Evaluate First
Don't rely on recipe conditionals to protect against missing data. Variables are evaluated before conditionals run.
2. Use "Option A" Pattern
Move conditionals inside bash commands. Ensure every step returns output, even if empty.
3. Handle Large Files with Stdin
Use stdin piping (echo "$VAR" | curl --data-binary @-) to avoid ARG_MAX limits.
4. Optimize for Local-First
When processing bulk data, prefer filesystem reads over API calls. Orders of magnitude faster.
