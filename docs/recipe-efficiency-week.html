Document Generation Recipe v7.0 → v8.0
The Challenge

"25 minutes is
too long"

Brian's feedback after the document generation demo drove this week's mission: make recipes fast, smart, and efficient.

Slow Execution

Sequential steps meant long wait times, even for simple document updates

Uniform Models

Every step used the same model, regardless of complexity

Full Regeneration

No way to leverage existing documents - always started from scratch

The Solution

Three-Pronged
Optimization

Parallel Processing

Validation checks now run concurrently. Content and quality checks execute in parallel groups.

Smart Model Selection

Haiku for parsing, Sonnet for analysis, Opus for generation. Right model for each job.

Existing Doc Support

Feed in existing documents. Preserve good content, revise outdated, replace poor.

Velocity

8 Versions
in 2 Days

v7.1 Existing document input support
v7.2 Deterministic preservation - no LLM for unchanged sections
v7.3 Comprehensive analysis - content AND quality in one pass
v7.4 Deterministic section restore - 100% preservation guarantee
v7.5 Code block aware parsing - fixed content truncation bug
v7.x Per-step model selection + provider fallback chains
v8.0 Merged release - all features combined

Key Innovations

What Makes
It Work

  • Deterministic preservation: Sections with no changes bypass the LLM entirely
  • Provider fallback chains: Haiku → GPT-4o-mini, Sonnet → GPT-4o, resilient across configs
  • Source-driven analysis: Only flag changes when sources actually indicate them
  • Code block aware parsing: No more content truncation from # in code blocks
