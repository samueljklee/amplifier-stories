Understanding the composition model

January 2026
Bundles

Composable configuration packages

A bundle combines modules (providers, tools, hooks), context files (instructions, knowledge), and agent definitions into a reusable package.
  • → Configuration, not code
  • → Markdown + YAML frontmatter format
  • → Can include other bundles
  • → Version controlled like code
Design Principle

Bundles compose, they don't implement

✓ Good Bundle
Includes foundation bundle, adds context files, defines agent personas. Pure composition.
✗ Bad Bundle
Implements Python code, duplicates module functionality. Should be a module instead.
The test: If it has Python code implementing capabilities, it's a module. If it's YAML/markdown composing capabilities, it's a bundle.
Anatomy

What's inside a bundle?

--- YAML frontmatter --- bundle: name: my-bundle version: 1.0.0 description: What this bundle provides includes: - bundle: foundation # Include other bundles - bundle: my-bundle:behaviors/main --- Markdown content --- # Instructions and context @my-bundle:context/instructions.md @foundation:context/common-system-base.md
Agents

Agents use bundle format

Key insight: Agents use the same file format (markdown + YAML) as bundles but are configuration overlays for specialized personas. Frontmatter uses meta: instead of bundle:
--- Agent definition --- meta: name: zen-architect description: Code planning and architecture specialist --- Agent-specific context --- # Zen Architect Instructions You embody ruthless simplicity...
Agent Mechanics

Session spawning

Parent Session
↓ task tool
Child Session (Agent)
↓ returns result
Parent Session
  • → Child runs its own orchestrator loop
  • → Can inherit context from parent
  • → Returns single result message
  • → Linked via parent_id for tracing
