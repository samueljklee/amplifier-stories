The Problem

Planning and implementing don't mix

Sometimes you want to think through a problem before writing code. But the assistant doesn't know that.

Premature Implementation
You ask about architecture and get 500 lines of code. You wanted a discussion, not a commit.
Manual Reminders
"Don't write any files yet" repeated in every message. Tedious and easy to forget.
No Feedback
No visual indication of current operating mode. Is the assistant planning or implementing?

Available Modes

Three modes for different workflows

/plan
Think and discuss, don't implement. Perfect for architecture discussions, design reviews, and problem decomposition.
/review
Analyze and critique code without modifying. Read files, run tests, but no changes. Pure code review mode.
/explore
Understand before acting. Investigate the codebase, trace dependencies, gather context before making decisions.
Interface

Simple commands

/modes
List all available modes with descriptions
/mode <name>
Enable a specific mode (e.g., /mode plan)
/mode off
Disable the current mode and return to normal
/plan
Quick shortcut to enable plan mode
Modes are toggle-based: one command on, one command off. No complex state to manage.
Architecture

Three-layer enforcement

1

Context Injection

prompt:submit hook injects <system-reminder> each turn with mode-specific guidance

2

Tool Moderation

tool:pre hook applies allow/warn/block policy per tool based on active mode

3

State Management

Mode state propagates to child sessions, maintaining consistent behavior

Enforcement

Tool policies in plan mode

Safe
read_file, glob, grep, web_search, task, todo
Always allowed. Reading and thinking tools work normally.
Warn First
bash
Warned once, then allowed. Useful for exploration commands.
Blocked
write_file, edit_file
Denied with explanation. No file modifications in plan mode.
Warn-first pattern: bash is useful for exploration (git log, ls, etc). Warn once, allow after acknowledgment.

Extensible

Add custom modes via markdown

amplifier-bundle-modes/ ├── bundle.md # Thin wrapper ├── behaviors/modes.yaml # Hook configuration ├── modes/ # Mode definitions │ ├── plan.md # Planning mode │ ├── review.md # Code review mode │ └── explore.md # Exploration mode └── modules/hooks-mode/ # Generic mode engine
Create your own modes: Add a markdown file to the modes/ directory. Define context injection text and tool policies. The generic engine handles the rest.
